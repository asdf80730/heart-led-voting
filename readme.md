# 一、系統架構

```text
LINE LIFF
   ↓
GitHub Pages 前端
   ↓ HTTPS API
Google Apps Script 後端
   ↓
Google 試算表
```

| 元件 | 技術 | 職責 |
|---|---|---|
| 前端 | GitHub Pages | LIFF 初始化、登入、畫面顯示、呼叫 API |
| 身分驗證 | LINE LIFF ID Token | 識別使用者 |
| 後端 | Google Apps Script | 驗證 Token、權限判斷、投票邏輯 |
| 資料庫 | Google 試算表 | 儲存系統設定、使用者、投票及紀錄 |
| 管理介面 | Google 試算表及 GAS 編輯器 | 管理者維護資料及新增投票 |

Google Apps Script 不提供前端 HTML，而是作為 API 後端使用。

LINE 官方 Verify ID Token API 負責驗證 LIFF 取得的 ID Token。

# 二、認證與密鑰規格

## 2.1 前端公開設定

GitHub Pages 前端可以放置：

```text
LIFF_ID
GAS_API_URL
```

這些資料不是秘密資料，可以出現在公開 GitHub Repository。

## 2.2 GAS Script Properties

GAS 僅需保存：

```text
CHANNEL_ID
SPREADSHEET_ID
```

| 設定 | 用途 |
|---|---|
| `CHANNEL_ID` | 傳送至 LINE 官方 Token 驗證 API |
| `SPREADSHEET_ID` | 指定資料試算表 |

## 2.3 不使用的密鑰

本系統不使用：

```text
API_SECRET
CHANNEL_SECRET
CHANNEL_ACCESS_TOKEN
```

### `API_SECRET`

`API_SECRET` 是自行設計的 API 密鑰，不是 LIFF 或 GAS 的必要設定。本系統不採用。

### `CHANNEL_SECRET`

本系統不自行驗證 JWT 簽章，而是由 GAS 將下列資料交給 LINE 官方驗證：

```text
id_token
client_id：CHANNEL_ID
```

因此不保存 `CHANNEL_SECRET`。

### `CHANNEL_ACCESS_TOKEN`

本系統不使用 Messaging API 或其他需要 Channel Access Token 的功能，因此不保存 `CHANNEL_ACCESS_TOKEN`。

## 2.4 重要限制

任何真正的秘密都不得放在：

- GitHub Pages。
- `index.html`。
- 公開 GitHub Repository。
- 前端 JavaScript。
- 前端設定檔。

# 三、LIFF 設定

## 3.1 LIFF Endpoint URL

LIFF Endpoint URL 必須設定為 GitHub Pages 實際網址：

```text
https://帳號.github.io/專案名稱/
```

不得設定為 GAS Web App URL：

```text
https://script.google.com/macros/s/部署ID/exec
```

目前執行 `liff.init()` 的頁面必須：

- 與 LIFF Endpoint URL 相同；或
- 位於 LIFF Endpoint URL 之下層路徑。

## 3.2 前端登入流程

```text
載入 GitHub Pages
→ 執行 liff.init()
→ 判斷是否登入
→ 未登入時執行 liff.login()
→ 使用 liff.getIDToken() 取得 ID Token
→ 傳送 ID Token 至 GAS
→ GAS 向 LINE 官方驗證
→ 回傳使用者權限狀態
```

前端不可使用自行傳入的 LINE User ID 作為權限依據。

## 3.3 GAS 驗證流程

```text
收到 ID Token
→ 呼叫 LINE Verify ID Token API
→ 使用 CHANNEL_ID 驗證
→ 取得經驗證的 LINE User ID
→ 查詢使用者工作表
→ 建立或判斷使用者狀態
```

`liff.getProfile()` 或前端自行傳送的顯示名稱：

- 可以作為第一次建立使用者時的輔助資料。
- 不得作為後端授權依據。
- 不得覆蓋既有使用者顯示名稱。

# 四、前端規格

## 4.1 前端負責功能

前端負責：

- LIFF 初始化。
- LINE 登入。
- 取得 ID Token。
- 顯示登入載入畫面。
- 顯示未授權畫面。
- 顯示停用畫面。
- 顯示啟用中的投票。
- 顯示投票題目及選項。
- 顯示各選項目前票數。
- 送出投票。
- 修改投票。
- 新增投票選項。
- 顯示投票結果。
- 執行瀏覽器列印。
- 顯示錯誤訊息。
- 顯示投票成功或失敗訊息。

## 4.2 前端不可自行決定

以下內容必須由後端判斷：

- 使用者是否存在。
- 使用者是否啟用。
- 投票是否啟用。
- 是否已截止。
- 使用者是否在黑名單。
- 是否超過選項上限。
- 投票是否成功。
- 投票票數。
- 投票人數。
- 使用者自己的投票紀錄。

前端隱藏按鈕、停用按鈕或畫面限制，不能視為安全控管。

## 4.3 投票頁面顯示結果

【本次修訂】

使用者進入單筆投票頁面時，必須同時看到：

- 投票題目。
- 投票選項。
- 單選或複選控制項。
- 截止日期。
- 自己的投票狀態。
- 每個選項的目前票數。

票數顯示於各選項最右側。

範例：

```text
○ 方案 A                                      12 票
○ 方案 B                                       8 票
○ 方案 C                                       3 票
```

複選範例：

```text
☑ 方案 A                                      12 票
☐ 方案 B                                       8 票
☐ 方案 C                                       3 票
```

票數必須由後端提供，前端不得自行計算。

## 4.4 投票頁面重新取得資料

【本次修訂】

每次進入單筆投票頁面時，前端都必須重新呼叫：

```text
getVote
```

以取得最新：

- 投票選項。
- 各選項票數。
- 投票人數。
- 自己的投票紀錄。
- 投票截止狀態。
- 投票是否仍啟用。

前端不得使用舊的結果快取取代 API 查詢。

# 五、Google Apps Script 後端規格

## 5.1 GAS Web App

GAS 部署後取得 API URL：

```text
https://script.google.com/macros/s/部署ID/exec
```

此網址只作為 API 使用，不作為 LIFF Endpoint URL。

## 5.2 API 入口

```javascript
function doGet(e) {
  return 處理API請求_(e, 'GET');
}

function doPost(e) {
  return 處理API請求_(e, 'POST');
}
```

兩者皆：

- 回傳 JSON。
- 不回傳 `HtmlService`。
- 不提供前端 HTML。
- 使用統一 API 回應格式。

## 5.3 API 統一回應格式

成功：

```json
{
  "ok": true,
  "data": {},
  "error": null,
  "meta": {}
}
```

失敗：

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "ERROR_CODE",
    "message": "錯誤說明"
  },
  "meta": {}
}
```

`meta` 可以包含：

- `requestId`
- `durationMs`
- 其他非敏感診斷資訊

不得在回應中包含：

- `CHANNEL_SECRET`
- Token 內容。
- 其他使用者的敏感資料。
- Script Properties。
- 黑名單完整資料。

## 5.4 API 動作

| 動作 | 方法 | 用途 |
|---|---|---|
| `bootstrap` | `POST` | 初始化使用者狀態及啟用投票列表 |
| `getSession` | `POST` | 驗證登入及取得權限 |
| `getVotes` | `POST` | 取得啟用中的投票 |
| `getVote` | `POST` | 取得單筆投票、自己的紀錄及目前結果 |
| `submitVote` | `POST` | 新增或更新投票 |
| `addOption` | `POST` | 新增投票選項 |
| `getPrintData` | `POST` | 取得列印用的結果及完整明細 |

## 5.5 移除的 API

【本次修訂】

不再提供：

```text
getResult
getDetails
```

原因：

- `getResult` 的功能併入 `getVote`。
- `getDetails` 不再提供線上明細頁面。
- 完整明細只在 `getPrintData` 中提供，作為列印內容的一部分。

# 六、跨來源請求規格

GitHub Pages 與 GAS 屬於不同來源，因此必須進行跨來源測試。

## 6.1 第一階段採用方式

- 查詢及寫入均可使用 `POST`。
- 將資料放在 request body。
- ID Token 放在 request body。
- 不使用自訂 `Authorization` Header。
- 不使用不必要的自訂 Header。
- 前端以簡單格式傳送請求。
- 實際測試 LINE 內建瀏覽器、iOS 及 Android。

前端可以使用：

```javascript
new URLSearchParams({
  payload: JSON.stringify(request)
})
```

後端必須正確解析 `application/x-www-form-urlencoded`。

## 6.2 表單資料解析

後端解析表單資料時，必須正確處理 `+` 代表空白的情況。

例如：

```text
方案 A
```

不得被錯誤解析為：

```text
方案+A
```

建議使用：

```javascript
function 解碼表單值_(value) {
  return decodeURIComponent(
    String(value || '').replace(/\+/g, ' ')
  );
}
```

## 6.3 CORS 驗收

必須確認：

- GitHub Pages 可以呼叫 GAS。
- GAS 可以回傳 JSON。
- 瀏覽器不阻擋回應。
- 不會因預檢請求而失敗。
- LINE 內建瀏覽器可以呼叫。
- iOS 可以呼叫。
- Android 可以呼叫。

若直接呼叫 GAS 無法穩定通過跨來源限制，再增加：

```text
GitHub Pages
→ Cloudflare Worker／Netlify Function／Firebase Functions
→ GAS
```

Proxy 只負責跨來源轉送：

- 不負責投票權限。
- 不負責使用者授權。
- 不負責投票邏輯。
- 不保存秘密資料。
- 投票權限及資料邏輯仍由 GAS 執行。

# 七、試算表工作表

系統使用四張工作表：

```text
系統設定
使用者
投票
投票紀錄
```

## 7.1 初始化規則

初始化時：

- 不刪除既有工作表。
- 不刪除既有資料。
- 不覆蓋既有設定。
- 不重複建立工作表。
- 不重複建立設定名稱。
- 標題欄位必須完全符合規格。
- 標題欄位順序必須完全符合規格。
- 標題欄位數量必須完全符合規格。
- 不得存在額外欄位。
- 標題不符時顯示錯誤並停止。

以下欄位不得出現在規格工作表中：

```text
選項上限
建立時間
修改時間
新增者
新增時間
API_SECRET
CHANNEL_SECRET
CHANNEL_ACCESS_TOKEN
DEFAULT_VOTE_STATUS
```

## 7.2 標題檢查規則

每張工作表必須檢查：

1. 實際欄位數量是否等於預期欄位數量。
2. 每個欄位名稱是否完全相同。
3. 每個欄位順序是否完全相同。
4. 是否有缺少欄位。
5. 是否有額外欄位。

任何一項不符，都必須停止初始化或 API 操作。

# 八、`系統設定` 工作表

## 8.1 欄位

```text
設定名稱
設定值
說明
是否必填
最後更新時間
```

## 8.2 設定項目

| 設定名稱 | 用途 | 預設值 |
|---|---|---|
| `TIMEZONE` | 日期判斷時區 | `Asia/Taipei` |
| `MAX_OPTIONS` | 所有投票共用的選項上限 | `10` |
| `VOTE_ID_FORMAT` | 投票 ID 格式 | `yyyyMM` |
| `SYSTEM_NAME` | 系統名稱 | `線上投票系統` |
| `DEFAULT_MARKDOWN_TITLE` | 新增投票預設題目 | 可留白 |
| `DEFAULT_OPTIONS` | 新增投票預設選項 | 可留白 |
| `DEFAULT_MULTI_SELECT` | 新增投票預設是否複選 | `否` |
| `DEFAULT_DEADLINE` | 新增投票預設截止日期 | 可留白 |

## 8.3 不包含的設定

不得包含：

```text
DEFAULT_VOTE_STATUS
API_SECRET
CHANNEL_SECRET
CHANNEL_ACCESS_TOKEN
```

新增投票的狀態固定由程式設定為：

```text
停用
```

## 8.4 驗證規則

| 設定 | 規則 |
|---|---|
| `MAX_OPTIONS` | 必須是大於零的整數 |
| `DEFAULT_MULTI_SELECT` | 只能是 `是` 或 `否` |
| `DEFAULT_DEADLINE` | 空白或符合 `YYYY-MM-DD` |
| `DEFAULT_OPTIONS` | 每行一個選項，不得超過 `MAX_OPTIONS` |
| `TIMEZONE` | 必須是有效時區 |
| 必填設定 | 必須存在且不可缺少 |
| `SYSTEM_NAME` | 必填時不可為空白 |
| `VOTE_ID_FORMAT` | 必填時不可為空白 |

設定錯誤時：

- 必須停止操作。
- 必須回傳錯誤。
- 不得猜測。
- 不得自動套用不明值。
- 不得靜默使用錯誤設定。

## 8.5 `DEFAULT_OPTIONS` 規則

`DEFAULT_OPTIONS`：

- 每行一個選項。
- 空白行不計算。
- 前後空白應移除。
- 完全相同的選項不得重複。
- 選項數量不得超過 `MAX_OPTIONS`。

- 狀態維持停用。
- 顯示名稱不更新。
- 備註不更新。
- 建立時間不更新。
- 不允許取得投票資料。
- 不允許投票。
- 不允許新增選項。
- 不允許取得列印資料。

## 9.6 使用者資料並發

自動建立使用者時必須：

1. 取得 `ScriptLock`。
2. 重新讀取使用者工作表。
3. 再次確認使用者是否已存在。
4. 只有確認不存在時才建立。
5. 釋放鎖定。

不得在鎖定前讀取一次，然後直接使用舊資料新增。

# 十二、投票 ID

格式範例：

```text
202607-001
```

規則：

- 程式自動產生。
- 不可重複。
- 管理者不需手動輸入。
- 使用 `LockService` 防止同時產生重複 ID。
- 依投票 ID 由新到舊排列。
- 編號依 `VOTE_ID_FORMAT` 產生前綴。
- 同一前綴下，序號取目前最大值加一。
- 序號至少使用三位數格式。

產生流程：

```text
取得 ScriptLock
→ 重新讀取投票工作表
→ 依當期前綴找出最大序號
→ 產生下一個序號
→ 寫入投票資料
→ 釋放 ScriptLock
```

# 十三、投票題目及選項

## 13.1 Markdown 題目

題目存放於單一儲存格。

前端顯示前：

1. 先將 HTML 特殊字元 escape。
2. 再轉換允許的 Markdown 格式。
3. 只允許規格定義的有限 Markdown。
4. 不得直接信任儲存格內的 HTML。
5. 不得執行 `<script>` 或事件處理器。
6. 不得產生危險連結或危險 HTML。

允許的 Markdown 可包含：

- `# 標題`
- `## 標題`
- `### 標題`
- `**粗體**`
- `*斜體*`
- `` `程式碼` ``

## 13.2 選項格式

```text
方案 A
方案 B
方案 C
```

規則：

- 每行一個選項。
- 空白行不計算。
- 前後空白移除。
- 完全相同的選項不得重複。
- 所有投票共用 `MAX_OPTIONS`。
- 使用者只能新增。
- 使用者不能修改既有選項。
- 使用者不能刪除既有選項。
- 新選項附加於最後。
- 不記錄新增者。
- 不記錄新增時間。

## 13.3 新增選項流程

```text
驗證 ID Token
→ 確認使用者啟用
→ 取得 ScriptLock
→ 重新讀取最新系統設定
→ 重新讀取最新投票資料
→ 確認投票啟用
→ 確認未截止
→ 確認不在黑名單
→ 重新解析最新選項
→ 檢查重複
→ 檢查 MAX_OPTIONS
→ 將新選項附加於最後
→ 寫入投票工作表
→ 釋放 ScriptLock
```

取得鎖定前讀取的投票資料不得直接用於寫入判斷。

## 13.4 新增選項並發要求

並發新增選項時：

- 不得覆蓋其他使用者剛新增的選項。
- 不得使用鎖定前的舊資料寫入。
- 必須在鎖定後重新讀取。
- 必須在鎖定後重新檢查重複。
- 必須在鎖定後重新檢查數量上限。

- 不得因重複提交建立第二筆相同紀錄。

## 17.4 投票紀錄欄位內容

| 欄位 | 格式 |
|---|---|
| 投票紀錄 ID | `投票ID::LINEUserID` |


因為紀錄 ID 格式為：

```text

```

所以可以從投票 ID 後方取得 LINE User ID。

此 fallback 必須由後端執行，前端不得自行信任或自行傳入 User ID。

# 十八、投票資料驗證

後端必須確認：

- 選項行號為有效整數。
- 選項行號沒有重複。
- 行號未超過有效選項數。
- 選項內容快照數量與行號一致。
- 單選只能選一個選項。
- 複選至少選一個選項。
- 投票者為啟用使用者。
- 投票狀態為啟用。
- 投票尚未截止。
- 投票者不在黑名單。
- 投票 ID 存在。
- 投票選項資料沒有異常。
- 快照內容與後端目前選項完全一致。

資料異常時必須：

- 拒絕操作。
- 不寫入任何資料。
- 回傳明確錯誤。
- 不猜測使用者意圖。
- 不自動修正選項行號。
- 不自動接受不一致的快照。

## 18.1 快照驗證

前端傳送：

```javascript
{
  voteId: '202607-001',
  selectedIndexes: [1, 3],
  snapshots: ['方案 A', '方案 C']
}
```

後端必須依目前最新投票選項確認：

```text
selectedIndexes[0] 對應 snapshots[0]
selectedIndexes[1] 對應 snapshots[1]
```

若選項在前端載入後已被新增或內容異常變更，必須拒絕並要求重新載入。
