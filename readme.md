
# GitHub Pages＋GAS 線上投票系統正式規格

> 本規格依目前提供的 `Code.gs`、`app.js`、`index.html` 及本次補充的後端程式檢查結果整理。  
> 「已完成」代表目前程式已有對應實作；「【待後續處理】」代表仍需修改或測試。  
> 本版本保留「投票後不重新呼叫 `getVote`」的效能設計，並納入 P1 前端操作狀態修正。

---

# 一、目前實作狀態說明

## 1.1 資料更新策略

投票及新增選項成功後，**不強制再次呼叫 `getVote`**。

目前流程：

- 投票或修改投票：

  `submitVote` → 後端寫入並重新計算結果 → 回傳完整最新 `vote` → 前端直接更新畫面

- 新增選項：

  `addOption` → 後端寫入新選項 → 回傳最新 `options` → 前端保留原有勾選並局部更新畫面

此設計用於避免重複讀取試算表、重新解析所有紀錄及重新計算結果造成過長等待時間。

## 1.2 仍會呼叫 `getVote` 的時機

- 使用者進入單筆投票頁面。
- 使用者主動重新開啟單筆投票。
- 使用者主動重新整理頁面或重新登入。
- 快照不一致時，前端要求使用者重新載入。

## 1.3 P1 前端狀態修正

目前前端已新增操作狀態 `state.operationBusy`，並透過 `同步投票操作狀態_()` 統一判斷：

- 投票按鈕。
- 新增選項按鈕。
- 列印按鈕。
- 投票選項控制項。
- 新增選項儲存按鈕。

操作完成後不會無條件重新啟用按鈕，而是重新依據目前投票狀態判斷。

# 二、全域規則

## 2.1 安全原則

- 前端只負責畫面、互動、LIFF 登入及 API 呼叫。
- 權限、狀態、截止日期、黑名單、選項數量、票數及結果均由 GAS 判斷。
- 前端停用或隱藏按鈕不屬於安全控管。
- 不信任前端傳入的 User ID、顯示名稱、權限、票數或黑名單狀態。
- 所有需要權限的 API 都必須重新驗證 LINE ID Token。
- 發生資料異常時，必須拒絕操作，不猜測、不自動修正、不靜默套用預設值。
- 後端不回傳完整黑名單或不必要的個人敏感資料。
- 後端 Log 不得記錄 ID Token、密鑰或完整投票內容。

## 2.2 鎖定原則

下列操作必須使用 `LockService`：

- 初始化系統。
- 新增使用者。
- 新增投票。
- 產生投票 ID。
- 新增選項。
- 新增或更新投票紀錄。

取得鎖定後必須：

1. 重新讀取最新設定及資料。
2. 重新驗證權限、狀態、截止日期及資料限制。
3. 重新檢查重複及資料一致性。
4. 完成寫入或更新。
5. 使用 `finally` 釋放鎖定。

不得使用鎖定前讀取的舊資料直接寫入。

## 2.3 日期規則

- 日期格式為 `YYYY-MM-DD`。
- 空白代表無截止限制。
- 截止日當天仍可投票。
- 超過截止日不可投票、修改投票或新增選項。
- 日期判斷及格式化使用 `系統設定.TIMEZONE`。
- 不使用使用者手機的日期或時區。
- 投票紀錄時間由 GAS 產生。
- 投票紀錄時間格式為 `yyyy-MM-dd HH:mm:ss`。

# 五、LIFF 與認證

## 5.1 Endpoint URL

LIFF Endpoint URL 必須是 GitHub Pages，例如：

`https://帳號.github.io/專案名稱/`

不得使用 GAS Web App URL。

## 5.2 登入流程

1. 載入 GitHub Pages。
2. 執行 `liff.init()`。
3. 判斷登入狀態。
4. 未登入時執行 `liff.login()`。
5. 取得 `liff.getIDToken()`。
6. 將 ID Token 傳送至 GAS。
7. GAS 呼叫 LINE Verify ID Token API。
8. 取得經驗證的 LINE User ID。
9. 查詢或建立使用者。
10. 回傳使用者授權狀態。

前端不得以自行傳入的 User ID 作為授權依據。

## 5.3 Token 驗證

GAS 使用 LINE Verify ID Token API：

`https://api.line.me/oauth2/v2.1/verify`

驗證內容包括：

- ID Token 是否存在。
- `CHANNEL_ID` 是否已設定。
- LINE API 回應狀態是否為成功。
- 回應是否包含 User ID。
- 由 LINE 驗證結果取得 User ID 及顯示名稱。

## 5.4 顯示名稱

顯示名稱只能：

- 作為新使用者建立時的輔助資料。
- 不得作為授權依據。
- 不得覆蓋既有使用者顯示名稱。
- 既有紀錄的顯示名稱保留於投票紀錄中。

# 六、前端規格

## 6.1 前端負責功能

前端負責：

- LIFF 初始化及登入。
- 取得 ID Token。
- 顯示載入、錯誤、未授權及停用畫面。
- 顯示啟用中的投票。
- 顯示題目、選項、票數、投票人數及截止狀態。
- 送出及修改投票。
- 新增選項。
- 顯示成功及錯誤訊息。
- 執行列印。
- 管理投票、新增選項及列印期間的操作狀態。

## 6.2 目前頁面

目前 HTML 提供：

- 載入畫面。
- 錯誤畫面。
- 尚未授權畫面。
- 帳號停用畫面。
- 投票列表。
- 單筆投票頁面。
- 返回列表。
- 投票及修改投票。
- 新增選項。
- 列印結果及明細。

目前沒有獨立的線上結果頁面或線上明細頁面。

## 6.3 單筆投票頁面

每次進入單筆投票頁面都呼叫 `getVote`。

頁面顯示：

- 系統名稱。
- 投票 ID。
- Markdown 題目。
- 所有有效選項。
- 單選或複選狀態。
- 截止日期。
- 投票是否截止。
- 各選項目前票數。
- 投票人數。
- 個人投票狀態。
- 個人首次投票時間及最後修改時間。

票數由後端提供，前端不得依畫面資料自行統計。

## 6.4 投票及修改投票

操作期間必須：

1. 顯示處理中訊息。
2. 停用送出按鈕。
3. 停用選項控制項。
4. 停用新增選項按鈕。
5. 停用列印按鈕。
6. 防止重複點擊及重複提交。
7. 成功後使用 `submitVote` 回傳的完整 `vote` 更新畫面。
8. 失敗後顯示錯誤。
9. 最後依目前投票狀態恢復正確操作狀態。

按鈕恢復時不得無條件啟用，必須重新判斷：

- `vote.closed`
- `vote.canVote`
- `vote.canAddOption`
- `vote.blacklisted`
- `state.operationBusy`

投票成功後目前**不再呼叫 `getVote`**。

## 6.5 `submitVote` 成功後流程

目前正式流程：

`送出投票`  
→ `submitVote`  
→ 後端取得鎖定  
→ 後端重新驗證並寫入  
→ 後端重新計算最新票數  
→ 回傳完整 `vote`  
→ 前端直接重新渲染  
→ 前端重新同步操作狀態

`submitVote` 的回傳資料是成功後的主要更新來源。

## 6.6 前端操作狀態

前端使用 `state.operationBusy` 表示目前是否有操作正在處理。

操作期間：

- `submit-vote-button` 停用。
- `add-option-button` 停用。
- `print-button` 停用。
- `vote-form-area` 內的投票選項停用。
- `save-option-button` 停用。
- 按鈕文字顯示處理中狀態。
- 不得重複提交。

操作完成後，前端透過 `同步投票操作狀態_()` 重新判斷。

已截止或無權限時，即使 `operationBusy` 恢復為 `false`，相關按鈕仍必須維持停用。

## 6.7 新增選項

新增選項成功後：

- 不得清除尚未送出的勾選。
- 新選項附加於最後。
- 舊選項票數維持。
- 新選項初始票數為零。
- 前端採用局部更新。
- 目前不重新呼叫 `getVote`。
- 成功後移除新增選項輸入區塊。
- 更新後重新判斷投票及新增選項按鈕狀態。

`addOption` 成功回傳：

```javascript
{
  voteId: voteId,
  options: newOptions
}
```

目前不回傳完整 `vote`、最新票數或最新投票人數。

新增選項期間必須：

- 顯示處理中訊息。
- 停用新增選項按鈕。
- 停用投票操作。
- 停用列印操作。
- 防止重複提交。

## 6.8 返回列表

返回列表時：

- 清除 `state.currentVote`。
- 重設 `state.operationBusy`。
- 移除新增選項輸入區塊。
- 清除投票操作訊息。
- 顯示投票列表畫面。

## 6.9 Markdown 安全

題目顯示前必須：

1. Escape HTML 特殊字元。
2. 只轉換允許的 Markdown。
3. 禁止任意 HTML。
4. 禁止 `script`、事件處理器及危險連結。

目前允許：

- `# 標題`
- `## 標題`
- `### 標題`
- `**粗體**`
- `*斜體*`
- `` `程式碼` ``

後端的 `Markdown轉安全HTML_()` 會先進行 HTML Escape，再轉換允許的格式。

## 6.10 前端快取

不得使用結果、明細或列印資料快取取代必要 API。

但以下情況屬於正式設計，不視為錯誤快取：

- 投票成功後使用 `submitVote` 回傳的完整資料。
- 新增選項成功後使用回傳的選項陣列局部更新。

# 七、GAS API

## 7.1 API 入口

- `doGet(e)`
- `doPost(e)`

兩者均回傳 JSON，不回傳 `HtmlService`。

## 7.2 API 回應格式

成功回應包含：

- `ok: true`
- `data`
- `error: null`
- `meta`

失敗回應包含：

- `ok: false`
- `data: null`
- `error.code`
- `error.message`
- `meta`

`meta` 可包含：

- `requestId`
- `durationMs`

不得回傳：

- ID Token。
- Script Properties。
- API 秘密。
- 完整黑名單。
- 不必要的其他使用者敏感資料。

## 7.3 API 動作

| 動作 | 用途 |
|---|---|
| `bootstrap` | 初始化使用者狀態及投票列表 |
| `getSession` | 驗證登入及取得權限 |
| `getVotes` | 取得啟用投票列表 |
| `getVote` | 取得單筆投票、統計及個人紀錄 |
| `submitVote` | 新增或更新投票 |
| `addOption` | 新增投票選項 |
| `getPrintData` | 取得投票結果及完整明細 |

不提供：

- `getResult`
- `getDetails`

`getVote` 整合投票結果及個人紀錄；完整明細只由 `getPrintData` 提供。

## 7.4 `bootstrap`

`bootstrap` 會：

1. 驗證 ID Token。
2. 查詢或建立使用者。
3. 取得系統設定。
4. 啟用使用者取得投票列表。
5. 停用使用者取得空的 `votes`。

回傳資料包含：

- `systemName`
- `user`
- `votes`

## 7.5 `getSession`

`getSession` 會：

1. 驗證 ID Token。
2. 查詢或建立使用者。
3. 回傳公開使用者資料。

回傳資料包含：

- `userId`
- `displayName`
- `status`
- `authorized`

## 7.6 `submitVote` 回傳資料

`submitVote` 成功後回傳：

- `voteId`
- `operation`
- `message`
- 最新完整 `vote`
- 最新 `myRecord`

`operation` 只能是：

- `create`
- `update`

回傳的 `vote` 包含：

- 投票基本資料。
- `counts`
- `voterCount`
- `myRecord`
- `canVote`
- `canAddOption`
- `closed`
- `blacklisted`

## 7.7 `addOption` 回傳資料

`addOption` 成功後回傳：

- `voteId`
- 最新完整 `options` 陣列

目前不重新計算或回傳：

- `counts`
- `voterCount`
- `myRecord`

## 7.8 `getPrintData` 回傳資料

`getPrintData` 成功後回傳：

- `systemName`
- `vote`
- `result`
- `details`

其中：

- `result` 是統計結果物件。
- `details` 是完整明細陣列。
- `details` 不再包裝於額外的 `{ voteId, details }` 物件中。

# 八、跨來源請求

## 8.1 請求格式

第一階段使用：

- `POST`
- ID Token 放在 request body。
- 不使用自訂 `Authorization` Header。
- 不使用不必要的自訂 Header。
- 可使用 JSON 或 `URLSearchParams` 傳送資料。

後端必須正確解析：

- JSON body。
- `application/x-www-form-urlencoded` body。
- 字串形式的 `payload`。

## 8.2 `+` 解碼

表單解析時，`+` 必須解碼為空白，避免表單內容錯誤。

## 8.3 CORS 測試

必須測試：

- GitHub Pages 呼叫 GAS。
- GAS 回傳 JSON。
- 無預檢失敗。
- LINE 內建瀏覽器正常。
- iOS 正常。
- Android 正常。

若直接呼叫不穩定，可使用 Cloudflare Worker、Netlify Function 或 Firebase Functions 作為轉送 Proxy。

Proxy 不負責：

- 權限判斷。
- 投票邏輯。
- 秘密保存。
- 資料保存。

- 初始化完成後執行 `SpreadsheetApp.flush()`。

`使用者` 必須保留 `建立時間`。

`投票` 不得包含：

- `建立時間`
- `修改時間`
- `新增者`
- `新增時間`

# 十、系統設定

## 10.1 欄位

- `設定名稱`
- `設定值`
- `說明`
- `是否必填`
- `最後更新時間`

## 10.2 設定項目

| 設定名稱 | 用途 | 預設值 |
|---|---|---|
| `TIMEZONE` | 日期及時間時區 | `Asia/Taipei` |
| `MAX_OPTIONS` | 所有投票共用選項上限 | `10` |
| `VOTE_ID_FORMAT` | 投票 ID 前綴格式 | `yyyyMM` |
| `SYSTEM_NAME` | 系統名稱 | `線上投票系統` |
| `DEFAULT_MARKDOWN_TITLE` | 新增投票預設題目 | 空白 |
| `DEFAULT_OPTIONS` | 新增投票預設選項 | 空白 |
| `DEFAULT_MULTI_SELECT` | 預設是否複選 | `否` |
| `DEFAULT_DEADLINE` | 預設截止日期 | 空白 |

## 10.3 驗證

- `MAX_OPTIONS` 必須是大於零的整數。
- `DEFAULT_MULTI_SELECT` 只能是 `是` 或 `否`。
- `DEFAULT_DEADLINE` 必須是空白或有效的 `YYYY-MM-DD`。
- `TIMEZONE` 必須是有效時區。
- 必填設定必須存在且非空白。
- `DEFAULT_OPTIONS` 不得超過 `MAX_OPTIONS`。
- 選項前後空白會被移除。
- 選項不可重複。
- 選項中不可包含空白行。
- 系統設定名稱不可重複。
- 額外設定名稱目前不會被拒絕。

> 目前程式對空白選項行採取「拒絕」，不是忽略。

> 目前程式不會拒絕額外設定名稱；若未來要求嚴格白名單，需另行實作。

# 十一、使用者

## 11.1 欄位

- `LINE User ID`
- `顯示名稱`
- `使用者狀態`
- `備註`
- `建立時間`

## 11.2 建立規則

新使用者：

- User ID：LINE 官方驗證取得。
- 顯示名稱：首次取得的名稱；無名稱時使用 User ID。
- 使用者狀態：`停用`。
- 備註：`系統自動建立，請管理者確認`。
- 建立時間：由 GAS 產生。

既有使用者的以下欄位不得由程式自動覆蓋：

- 顯示名稱。
- 使用者狀態。
- 備註。
- 建立時間。

新增使用者時：

1. 取得 Script Lock。
2. 重新讀取使用者工作表。
3. 確認 User ID 不存在。
4. 寫入新使用者。
5. 使用 `finally` 釋放鎖定。

## 11.3 使用者狀態

目前有效狀態：

- `啟用`
- `停用`

新使用者一律為 `停用`，由管理者手動啟用。

# 十二、使用者權限

## 12.1 停用使用者

停用使用者可以取得：

- 經驗證的 User ID。
- 顯示名稱。
- 使用者狀態。
- 授權狀態。

# 十五、投票選項

## 15.1 選項規則

- 不得超過 `MAX_OPTIONS`。

- 不記錄新增者及新增時間。

## 15.2 新增流程

2. 確認使用者啟用。
3. 取得鎖定。
4. 重新讀取設定及投票。
5. 驗證投票啟用、未截止且使用者不在黑名單。
6. 重新解析選項。
7. 檢查重複及數量上限。
8. 附加選項並寫入。
9. 執行 `SpreadsheetApp.flush()`。
10. 使用 `finally` 釋放鎖定。

# 十九、投票資料驗證

- 投票紀錄資料本身沒有異常。

`SNAPSHOT_MISMATCH`

並要求使用者重新載入。
